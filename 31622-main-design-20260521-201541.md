## Problem Statement

高等数学学生面临的核心问题不是"找不到学习资料"，而是**不知道自己哪里没真懂**。一个学生看完"极限定义"后能做课后题，但遇到需要真正理解 ε-δ 的证明题就卡住——因为他的理解停留在"越来越接近"的直觉层面，而非精确定义层面。传统课堂 1对50 模式下，老师即使发现学生没真懂也没有时间一对一追问到底。

目标：构建一个**苏格拉底式多智能体教练系统**，通过自然语言对话诊断学生的知识盲区，用追问逼迫学生直面自己"装懂"的概念，然后协同多个专业Agent生成针对该盲区的个性化学习资源，形成"诊断→追问→生成→评估→追踪"的完整教学闭环。

## What Makes This Cool

不是又一个"AI帮你生成学习资料"的工具。而是一个**能发现你哪不懂、追着问你直到你真的懂了、然后专门为你的盲区生成补救材料**的数学教练。

对评委的demo故事线：一个学生说"我觉得我懂极限了"→ 系统5分钟诊断发现他在"无穷小比较"上只是背了等价无穷小公式 → 苏格拉底追问"sin x 约等于 x 什么时候对，什么时候不对？" → 学生卡住 → 系统发现根因是"泰勒展开"这个概念没掌握 → 自动生成泰勒展开的针对性讲义+动画+练习 → 学生学完回来 → 再次追问，这次通过了 → 画像更新，学习路径自动调整。

评委看到的不是功能checklist，而是一个**认知转变的story arc**。

## Constraints

1. **比赛要求**：第十五届中国软件杯，科大讯飞出题。必须使用多智能体架构、科大讯飞Spark API、生成≥5种类型学习资源
2. **时间**：比赛截止前需交付可运行系统+7分钟演示视频+PPT+配套文档
3. **技术栈**：Python后端+TypeScript前端，Spark API作为主力LLM之一，DeepSeek辅助
4. **范围**：聚焦"极限与连续"一章（约5-8个核心概念点），做深做透
5. **质量要求**：数学准确性不可妥协（SymPy验证+知识库交叉校验），防幻觉机制

## Premises

1. 核心差异化在"诊断→追问→发现根因→生成针对性资源→追踪理解"闭环，而非单一苏格拉底对话
2. 聚焦"极限与连续"一章，做深而非做广——评委记住转变故事胜过功能数量
3. 多Agent架构服务于教学逻辑——不同Agent各有分工，协作产出的教学闭环单模型做不到
4. Demo说服力来自展示真实认知转变过程——一个story arc，而非功能列表
5. Spark API作为主力LLM之一，框架支持多模型路由，不锁定单一供应商

## Approaches Considered

### Approach A: 全链路闭环版（已选择）

完整LangGraph多Agent系统。6个专业Agent协作，FastAPI+Next.js全栈，ChromaDB知识库，Spark+DeepSeek双模型。

- 6个Agent：Profile Builder, Diagnostician, Socratic Coach, Resource Generator (Lecture/Exercise/MindMap/Reading), Assessor, Quality Gate
- 5+种资源：教学讲义、思维导图、分类型练习题、拓展阅读、代码可视化/动画脚本
- 完整"极限与连续"知识库（LaTeX-aware chunking + BGE-M3 embedding + 混合检索）
- 3维学习画像，动态更新

### Approach B: 对话诊断轻量版

Gradio单页应用，3Agent（诊断+追问+生成），2-3种资源类型。开发快但比赛"多智能体架构"得分点不足。

### Approach C: 深度追问极致版

语音+情绪检测+心流调控，极致追问体验。技术风险高（中文数学ASR可用性未知），且资源类型不足。

## Recommended Approach

**Approach A — 全链路闭环版**，理由：
- 覆盖比赛全部5项核心功能需求和45%的"功能实现"评分
- 多Agent架构本身就是技术亮点，评审认可度高
- "苏格拉底教练"叙事线贯穿整个架构，给系统注入灵魂

叠加Approach C的追问体验为加分项（流式输出思考过程、追问决策可视化、心流提示），在A的架构基础上做交互体验的精致打磨。

## Architecture Overview

```
User → FastAPI/SSE → Orchestrator (LangGraph Supervisor)
                         ├── Profile Builder Agent (画像构建)
                         ├── Diagnostician Agent (盲区探测)

```

## Orchestrator: Intent Routing State Machine

Orchestrator 基于 LangGraph Supervisor 实现，根据用户输入和当前状态路由到对应 Agent。

```
                    ┌─────────────────┐
                    │   INIT (入口)    │
                    └────────┬────────┘
                             │
                    ┌────────v────────┐
                    │ PROFILE_CHECK   │
                    │ 首次用户? 画像存在? │
                    └───┬─────────┬───┘
                        │ 新用户   │ 老用户
               ┌────────v──┐  ┌───v─────────┐
               │ BUILD     │  │ DIAGNOSE     │
               │ PROFILE   │  │ 诊断盲区      │
               └────┬──────┘  └───┬──────────┘
                    │             │
                    └──────┬──────┘
                           │
                  ┌────────v────────┐
                  │ COACH (追问)     │
                  │ 多轮苏格拉底追问   │
                  └───┬─────────┬───┘
                      │ 学生卡住  │ 学生通过
             ┌────────v──┐  ┌───v─────────┐
             │ GENERATE  │  │ ASSESS       │
             │ 生成补救资源 │  │ 评估+更新画像  │
             └────┬──────┘  └───┬──────────┘
                  │             │
                  └──────┬──────┘
                         │
                ┌────────v────────┐
                │ QUALITY_GATE    │
                │ 安全+数学校验    │
                └───┬─────────┬───┘
                    │ 未通过  │ 通过
           ┌────────v──┐  ┌───v─────────┐
           │ REGENERATE │  │ RESPOND     │
           │ (最多2次)   │  │ 返回用户     │
           └────────────┘  └─────────────┘
```

状态转换规则：
- **INIT → PROFILE_CHECK**：用户发送第一条消息
- **PROFILE_CHECK → BUILD_PROFILE**：新用户（无画像），Profile Builder 通过 3-5 轮简短对话建立 v1 画像
- **PROFILE_CHECK → DIAGNOSE**：老用户（有画像），直接进入诊断
- **DIAGNOSE → COACH**：Diagnostician 完成盲区定位
- **COACH → GENERATE**：学生多次追问后仍卡住（confidence < 0.3），触发资源生成
- **COACH → ASSESS**：学生通过追问（confidence > 0.7），进入评估
- **GENERATE → COACH**：学生学完资源后回到追问验证
- **ASSESS → RESPOND**：评估完成，更新画像，返回结果
- **QUALITY_GATE → REGENERATE**：内容未通过安全检查或数学验证（最多重试 2 次）
- **QUALITY_GATE → RESPOND**：内容通过，流式返回用户

## Socratic Coach: Questioning Algorithm

Socratic Coach 采用**先答后问 + 分层追问**策略。核心原则：学生问知识性问题时先给出正确答案，再通过追问检验真实理解；学生答错时先纠正再继续，不让学生在不知道自己错了的情况下继续对话。

**追问流程：**

```
L0: 直接回答 + 纠错门
  ├── 学生提问 → 先给出准确答案（含简要推导），再过渡到追问
  └── 学生回答错误 → 明确指出错误 + 给出正确答案 + 解释为什么错
      例："sinx/cosx 在 x→0 时极限是 0，不是 1。因为 sin0=0, cos0=1，所以 0/1=0。
           你记成 tanx→1 可能是和 lim(x→0) sinx/x = 1 搞混了——这是另一个重要极限。
           那我们来看看 sinx/x 这个极限..."
      然后继续 L1 追问，聚焦学生暴露出的薄弱点

L1: 概念复述 — "用自己的话解释一下为什么 tanx 在 x→0 时等于 0？"
  ↓ 学生回答正确
L2: 边界追问 — "如果 x=0.001，tanx 大约是多少？和 0 差了多少？"
  ↓ 学生回答正确
L3: 反例挑战 — "f(x)=tanx 在 x→π/2 附近有极限吗？为什么？"
  ↓ 学生回答正确 → 通过，进入 ASSESS
  ↓ 任何一层答错 → 回到 L0 纠错门，明确指出错误后再继续
  ↓ 任何一层卡住（不知道/沉默）→ 触发盲区分析
```

**纠错门规则（L0）：**
- 学生答案正确 → 简短确认（"对，tan0=0"），进入下一层追问
- 学生答案错误 → 三步纠错：①指出错误 ②给出正确答案+推导 ③诊断可能的混淆来源（如混淆了 sinx/x 和 tanx 的极限），然后聚焦混淆点追问
- 学生连续 2 次在同一概念上答错 → 不是追问不够，是前置概念缺失，触发 Diagnostician 根因分析

**卡住处理流程：**
1. 区分"答错"和"卡住"：答错先走 L0 纠错门；卡住（不知道/沉默）走盲区分析
2. 记录卡住的概念节点和追问层级
3. Diagnostician 分析根因（前置概念缺失 or 当前概念理解不足）
4. 前置缺失 → 回溯到前置概念，重新追问
5. 当前概念卡住 → 触发 Resource Generator 生成针对性补救材料
6. 学生确认学完后 → 从卡住层级重新追问

**难度自适应：**
- `cautious` 风格学生：降低追问跳跃幅度，每轮给更多提示
- `exploratory` 风格学生：适当提高反例难度，鼓励独立思考
- `impulsive` 风格学生：在 L2/L3 层加强边界追问，防止浅尝辄止

**心流调控（体验加分项）：**
- 连续答对 2 层 → 提升难度，给出更有挑战性的反例
- 连续卡住 2 次 → 降级到更基础的问题，避免挫败感
- 流式输出思考过程（"让我想想... 你说的这个其实有一个小问题"），模拟人类教练的思考节奏

## Error Handling & Degradation Strategy

| 故障场景 | 降级策略 | 用户感知 |
|----------|----------|----------|
| Spark API 超时(>15s) | 自动切换到 DeepSeek 重试，若仍失败返回预设提示 | "我再想想这个问题..." 转 L1 层简单追问 |
| Spark API 全部不可用 | 全部请求路由到 DeepSeek，Quality Gate 严格模式 | 无明显影响（双模型设计） |
| 模型全部不可用 | 返回预设的通用苏格拉底问题列表 + 静态资源推荐 | "当前服务繁忙，请稍后再试。这是为你准备的思考题..." |
| ChromaDB 检索异常 | 降级到关键词匹配（jieba 分词 + TF-IDF） | 资源质量略降，但不影响核心追问流程 |
| Resource Generator 失败 | 重试最多 2 次，若仍失败跳过该资源类型 | 少一种资源，其他资源正常生成 |
| SymPy 验证超时 | 5s 超时，跳过验证，标注"未自动验证" | 数学内容默认可信，用户可见提示 |
| SSE 连接断开 | 前端自动重连（exponential backoff, max 3 次） | 短暂重连，不丢消息（后端缓存未发送的事件） |
| 用户输入包含敏感/非数学内容 | Quality Gate 拦截，返回预设引导语 | "让我们回到极限这个话题吧！刚才你提到的问题..." |

## Knowledge Base: 极限与连续

```
极限与连续/
├── 1.1-数列极限 (定义, ε-N语言, 收敛性判别, 四则运算, 夹逼准则, 单调有界)
├── 1.2-函数极限 (定义, ε-δ语言, 左右极限, 无穷小与无穷大比较, 极限运算法则)
├── 1.3-两个重要极限 (lim sinx/x=1, lim (1+1/x)^x=e, 等价无穷小替换)
├── 1.4-连续性 (连续定义, 间断点分类, 闭区间连续函数性质: 有界/最值/介值/零点)
└── 1.5-极限的应用 (渐近线, 函数作图, 连续函数的应用)
```

约15-20个知识节点（聚焦极限部分核心概念），每个节点包含：定义(LaTeX)、定理、例题+详解、常见错误、前置依赖、关联概念。

## Learning Profile: 3 Dimensions (v1)

1. **知识掌握度**：每个知识点的掌握分数(0-1)+置信度，贝叶斯更新
2. **盲区图谱**：已发现的误解概念、错误类型(概念/计算/符号/逻辑/前置缺失)、关联矩阵
3. **学习行为**：会话时长、追问响应模式(cautious/exploratory/impulsive)、资源偏好(visual/textual/interactive)

动态更新：显式(对话诊断) + 隐式(行为观察)，贝叶斯更新知识掌握分数。v1聚焦最小可用画像，v2+扩展认知风格、学习目标等维度。

## Demo Script: 7-Minute Story Arc

**0:00-1:00** — 场景设定：一个学生在复习极限，系统通过简短对话建立画像
**1:00-3:00** — 诊断展开：Diagnostician发现学生在"等价无穷小替换"上只是记了公式
**3:00-5:00** — 苏格拉底追问：Socratic Coach追问"sin x ≈ x 什么时候成立？f(x)=0.1时还能替换吗？"，学生卡住，系统发现根因是"泰勒展开"概念缺失
**5:00-6:30** — 资源生成：多个Agent并行生成泰勒展开的讲义+思维导图+分层练习+动画脚本
**6:30-7:00** — 闭环展示：学生学完后再次追问通过，画像自动更新，学习路径调整

## Success Criteria

1. 完整诊断→追问→生成→评估闭环可用
2. 生成≥5种类型学习资源，每种有明确的教学目的
3. 极限与连续核心知识库(≥15节点)，检索精度≥80%
4. 数学内容经SymPy验证无事实错误
5. 7分钟demo流畅展示完整story arc
6. 支持流式输出，无长时间白屏等待

## What I noticed about how you think

- 你选择了"苏格拉底教练"而非"资源生成工具"——说明你要的不是checklist式的功能堆砌，而是一个有教学哲学的产品。评委能感受到这种不同
- "一章做透"的选择很有胆量——大多数人不敢做减法，担心功能不够多。但deep > broad在竞赛评审中往往是赢家策略，因为评委记住的是体验而非功能数量
- 你把手伸向"情绪感知+心流"这个10x版本，但不执着于现在实现——知道北极星在哪，也知道第一步迈多远，这是工程判断力
- 当我说"vs ChatGPT/数学老师"时你选了vs数学老师——说明你对标的是人类教学体验，而不是另一个AI工具，这让产品定位天然高于竞品
