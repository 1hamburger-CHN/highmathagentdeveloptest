## Problem Statement

高等数学学生面临的核心问题不是"找不到学习资料"，而是**不知道自己哪里没真懂**。一个学生看完"极限定义"后能做课后题，但遇到需要真正理解 ε-δ 的证明题就卡住——因为他的理解停留在"越来越接近"的直觉层面，而非精确定义层面。传统课堂 1对50 模式下，老师即使发现学生没真懂也没有时间一对一追问到底。

目标：构建一个**苏格拉底式多智能体教练系统**，通过自然语言对话诊断学生的知识盲区，用追问逼迫学生直面自己"装懂"的概念，然后协同多个专业Agent生成针对该盲区的个性化学习资源，形成"诊断→追问→生成→评估→追踪"的完整教学闭环。

## What Makes This Cool

不是又一个"AI帮你生成学习资料"的工具。而是一个**能发现你哪不懂、追着问你直到你真的懂了、然后专门为你的盲区生成补救材料**的数学教练。

对评委的demo故事线：一个学生说"我觉得我懂极限了"→ 系统5分钟诊断发现他在"无穷小比较"上只是背了等价无穷小公式 → 苏格拉底追问"sin x 约等于 x 什么时候对，什么时候不对？" → 学生卡住 → 系统发现根因是"泰勒展开"这个概念没掌握 → 自动生成泰勒展开的针对性讲义+动画+练习 → 学生学完回来 → 再次追问，这次通过了 → 画像更新，学习路径自动调整。

评委看到的不是功能checklist，而是一个**认知转变的story arc**。

## Constraints

1. **比赛要求**：第十五届中国软件杯，科大讯飞出题。必须使用多智能体架构、科大讯飞Spark API、生成≥5种类型学习资源
2. **时间**：比赛截止前需交付可运行系统+7分钟演示视频+PPT+配套文档
3. **技术栈**：Python后端+TypeScript前端，Spark API作为主力LLM之一，DeepSeek辅助
4. **范围**：聚焦"极限与连续"一章（约5-8个核心概念点），做深做透
5. **质量要求**：数学准确性不可妥协（SymPy验证+知识库交叉校验），防幻觉机制

## Premises

1. 核心差异化在"诊断→追问→发现根因→生成针对性资源→追踪理解"闭环，而非单一苏格拉底对话
2. 聚焦"极限与连续"一章，做深而非做广——评委记住转变故事胜过功能数量
3. 多Agent架构服务于教学逻辑——不同Agent各有分工，协作产出的教学闭环单模型做不到
4. Demo说服力来自展示真实认知转变过程——一个story arc，而非功能列表
5. Spark API作为主力LLM之一，框架支持多模型路由，不锁定单一供应商

## Approaches Considered

### Approach A: 全链路闭环版（已选择）

完整LangGraph多Agent系统。9个专业Agent协作，FastAPI+Next.js全栈，ChromaDB知识库，Spark+DeepSeek双模型。

- 9个Agent：Orchestrator, Profile Builder, Diagnostician, Socratic Coach, Lecture Note, Exercise, Mind Map, Reading, Assessment（Quality Gate作为管线过滤器，不计入Agent数）
- 5+种资源：教学讲义、思维导图、分类型练习题、拓展阅读、代码可视化/动画脚本
- 完整"极限与连续"知识库（LaTeX-aware chunking + BGE-M3 embedding + 混合检索）
- 3维学习画像（v1精简版），动态更新

### Approach B: 对话诊断轻量版

Gradio单页应用，3Agent（诊断+追问+生成），2-3种资源类型。开发快但比赛"多智能体架构"得分点不足。

### Approach C: 深度追问极致版

语音+情绪检测+心流调控，极致追问体验。技术风险高（中文数学ASR可用性未知），且资源类型不足。

## Recommended Approach

**Approach A — 全链路闭环版**，理由：
- 覆盖比赛全部5项核心功能需求和45%的"功能实现"评分
- 多Agent架构本身就是技术亮点，评审认可度高
- "苏格拉底教练"叙事线贯穿整个架构，给系统注入灵魂

叠加Approach C的追问体验为加分项（流式输出思考过程、追问决策可视化、心流提示），在A的架构基础上做交互体验的精致打磨。

## Orchestrator: Intent Routing State Machine

Orchestrator是LangGraph Supervisor的入口节点，负责将用户输入路由到正确的Agent子图。状态机定义如下：

```
                    ┌─────────────┐
                    │   START     │
                    └─────┬───────┘
                          ↓
              ┌───────────────────────┐
              │   Intent Classifier   │
              │   (Spark Lite, ~50ms) │
              └───────┬───────────────┘
                      │
        ┌─────────────┼─────────────┐
        ↓             ↓             ↓
┌───────────┐  ┌───────────┐  ┌──────────────┐
│ DIAGNOSE  │  │  COACH    │  │   GENERATE   │
│  (诊断)   │  │  (追问)   │  │   (资源生成) │
└─────┬─────┘  └─────┬─────┘  └──────┬───────┘
      │              │              │
      ↓              ↓              ↓
┌───────────┐  ┌───────────┐  ┌──────────────┐
│Diagnostician│ │ Socratic  │  │ Fan-out to   │
│  Agent     │  │  Coach    │  │ 4 Resource   │
│            │  │  Agent    │  │   Agents     │
└─────┬─────┘  └─────┬─────┘  └──────┬───────┘
      │              │              │
      └──────────────┼──────────────┘
                     ↓
           ┌─────────────────┐
           │ ASSESS / PROFILE│
           │ (评估+画像更新) │
           └────────┬────────┘
                    ↓
           ┌─────────────────┐
           │   NEXT ACTION   │
           │ (Orchestrator   │
           │  决定下一步)    │
           └────────┬────────┘
                    │
        ┌───────────┼───────────┐
        ↓           ↓           ↓
   ┌────────┐ ┌────────┐ ┌────────┐
   │RETURN  │ │CONTINUE│ │ END    │
   │TO USER │ │COACH   │ │SESSION │
   └────────┘ └────────┘ └────────┘
```

**意图分类标签**:
| Intent | 触发条件 | 路由目标 |
|--------|---------|---------|
| `DIAGNOSE` | 首次会话、"开始诊断"、系统主动触发 | Diagnostician |
| `COACH` | 诊断完成后自动流转、用户回答问题后 | Socratic Coach |
| `GENERATE` | 追问发现根因后自动触发 | 4个Resource Agent fan-out |
| `ASSESS` | 用户提交答案、请求评估 | Assessment Agent |
| `CHAT` | 闲聊、疑问 | 直接LLM回复（不经过Agent） |
| `END` | 用户请求结束、超时 | Profile Builder更新画像后结束 |

**状态转换决策**: Orchestrator根据当前`LearningProfile`和对话历史决定下一步。诊断完成后自动进入追问、追问发现根因后自动触发资源生成、资源生成后返回追问验证——形成自动流转闭环，无需用户手动切换。

## Socratic Coach: Questioning Algorithm

Socratic Coach是系统的核心差异化引擎。采用**分层追问策略**，基于Bloom认知层次递进：

```
追问深度层级 (Bloom Taxonomy adapted for math):
┌────────────────────────────────────────────┐
│ L5 — 创造 (Create)                         │
│ "你能构造一个满足A但不满足B的反例吗？"     │
├────────────────────────────────────────────┤
│ L4 — 评价 (Evaluate)                       │
│ "为什么这个解法比那个更好？什么情况下失效？"│
├────────────────────────────────────────────┤
│ L3 — 分析 (Analyze)                        │
│ "证明sin x ~ x需要用到什么定理？条件是什么？"│
├────────────────────────────────────────────┤
│ L2 — 理解 (Understand)                     │
│ "用自己的话解释为什么lim sin x/x = 1成立"  │
├────────────────────────────────────────────┤
│ L1 — 记忆 (Remember)                       │
│ "等价无穷小替换的公式是什么？"             │
└────────────────────────────────────────────┘
```

**追问算法**:
1. **入口**: 从L2层开始提问（跳过L1纯记忆层，除非Diagnostician发现基础缺失）
2. **递进规则**: 学生连续2次正确回答 → 提升一层；学生卡住或错误 → 标记当前层为"边界层"
3. **根因定位**: 当学生在L2层卡住时，系统推断"概念理解有问题"；L3层卡住时推断"前置知识(泰勒展开)缺失"；L4层卡住时推断"应用迁移能力不足"
4. **话题生成策略**: 从知识库中检索"当前卡住概念"的所有前置依赖，按prerequisite链追溯到根因；同时检索该概念相关的"常见错误"节点，匹配学生的错误模式
5. **退出门槛**: 学生在L3层连续2次正确回答 → 追问通过；或追问轮次超过10轮 → 触发资源生成作为补救

**追问模板示例（极限/无穷小比较）**:
| 层次 | 示例问题 |
|------|---------|
| L2 | "sin x和x在x→0时谁趋近0更快？你能直观解释吗？" |
| L3 | "证明sin x ~ x需要洛必达法则还是泰勒展开？两种方法各有什么优缺点？" |
| L4 | "什么时候不能用等价无穷小替换？f(x)如果震荡很大还能替换吗？" |
| L5 | "构造一个函数，它在0附近和sin x行为类似但不满足等价无穷小条件" |

## Knowledge Base: 极限与连续

```
极限与连续/
├── 1.1-数列极限 (定义, ε-N语言, 收敛性判别, 四则运算, 夹逼准则, 单调有界)
├── 1.2-函数极限 (定义, ε-δ语言, 左右极限, 无穷小与无穷大比较, 极限运算法则)
├── 1.3-两个重要极限 (lim sinx/x=1, lim (1+1/x)^x=e, 等价无穷小替换)
├── 1.4-连续性 (连续定义, 间断点分类, 闭区间连续函数性质: 有界/最值/介值/零点)
└── 1.5-极限的应用 (渐近线, 函数作图, 连续函数的应用)
```

约15-20个知识节点（v1聚焦极限部分，连续性问题通过向量检索兜底），每个节点包含：定义(LaTeX)、定理、例题+详解、常见错误、前置依赖、关联概念。

## Learning Profile: 3 Dimensions (v1)

v1精简为3个核心维度，v2再扩展至7维：

1. **知识掌握度**：每个知识点的掌握分数(0-1)+置信度，贝叶斯更新
2. **易错点画像**：错误类型(概念/计算/符号/逻辑/前置缺失)、频率统计
3. **学习历史**：已完成主题、时间投入、资源使用记录

动态更新：显式(对话诊断) + 隐式(行为观察)，贝叶斯更新知识掌握分数。

> v2扩展维度：认知风格、学习目标、学习行为、数学基础。当前聚焦比赛demo可展示的最小闭环。

## Demo Script: 7-Minute Story Arc

**0:00-1:00** — 场景设定：一个学生在复习极限，系统通过简短对话建立画像
**1:00-3:00** — 诊断展开：Diagnostician发现学生在"等价无穷小替换"上只是记了公式
**3:00-5:00** — 苏格拉底追问：Socratic Coach追问"sin x ≈ x 什么时候成立？f(x)=0.1时还能替换吗？"，学生卡住，系统发现根因是"泰勒展开"概念缺失
**5:00-6:30** — 资源生成：多个Agent并行生成泰勒展开的讲义+思维导图+分层练习+动画脚本
**6:30-7:00** — 闭环展示：学生学完后再次追问通过，画像自动更新，学习路径调整

## Success Criteria

1. 完整诊断→追问→生成→评估闭环可用
2. 生成≥5种类型学习资源，每种有明确的教学目的
3. 极限知识库(≥15节点)，检索精度≥80%
4. 数学内容经SymPy验证无事实错误
5. 7分钟demo流畅展示完整story arc
6. 支持流式输出，无长时间白屏等待

## Error Handling & Degradation Strategy

| 故障场景 | 降级策略 | 用户感知 |
|---------|---------|---------|
| Spark API超时(>30s) | 自动重试1次，仍失败→切换DeepSeek | "正在切换备用模型，请稍候..." |
| Spark API全部不可用 | 降级至DeepSeek单模型模式 | 功能正常但追问质量可能略降 |
| 所有LLM不可用 | 返回预设错误提示+建议稍后重试 | "系统繁忙，请稍后再试" |
| ChromaDB检索失败 | 跳过RAG增强，仅用Agent内置知识 | 数学准确性可能下降，但对话不中断 |
| SymPy验证失败 | 标记"未验证"但不禁用资源生成 | 资源页显示"数学验证暂不可用"标签 |
| Fan-out部分Agent失败 | 已成功的Agent结果正常返回，失败的不展示 | 部分资源缺失，但不影响整体流程 |
| 单次对话超50轮 | 自动总结上下文，压缩历史 | 无感知 |
| 用户输入非数学内容 | 意图分类→CHAT，直接LLM回复 | 正常对话体验 |

**全局防护**:
- 所有Agent输出经过`Quality Gate`管线：数学一致性检查 + 内容安全过滤
- Spark API请求携带`temperature=0.3`（数学场景需低随机性），DeepSeek备用同样温控
- 知识库检索作为数学权威源，Agent生成内容与知识库冲突时以知识库为准

## Testing Strategy

### 分层测试金字塔

